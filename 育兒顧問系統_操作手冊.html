育兒顧問 AI 系統

從零開始的完整操作手冊 — 給沒寫過程式的你

預估 3–4 週（每天 1 小時）· 5 個 Phase

這份手冊是寫給你自己看的，不是給 Claude Code 的技術文件。每一步都假設你沒有程式背景，會告訴你「在哪裡、貼什麼、看到什麼算過關」。Claude Code 看的是 docs/ 裡那六份 .md 檔案。
目錄 系統長什麼樣 你的顧問團隊（9 個角色） 育兒管家：你說話，對的人出現 開始前的準備工作 Phase 0 — 骨架接軌 Phase 1 — 情報管線 Phase 2 — 月齡發展 + 管家上線 Phase 3 — 顧問團上線 Phase 4 — 收尾部署 總覽表

系統長什麼樣

完成後，你的 Slack 裡會有一個 #parenting-room 頻道。你在裡面說話就好——不需要記任何人的名字，不需要打 @ 符號，系統會自動判斷該叫誰來回應你。

它會做三件事：

主動推給你 — 每週一份育兒政策、社群話題、兒童安全的摘要，重大事件即時推送
每月問你 — 寶寶這個月發展到哪了？自動記錄你的回覆，追蹤里程碑
隨時問它 — 你說話就好，對的顧問會出現，回覆會附上參考來源

所有資訊都有來源分級（T1 官方 → T4 不明），醫療建議只取 T1/T2。社群說法永遠標記為「有些父母分享…」，不會被當成建議。

你的顧問團隊

9 個角色分三層——你平常只會看到顧問層（綠色的那些人），情報層和系統層在背後默默運作。

情報層（你看不到，默默蒐集資訊）

輿情蒐集秘書 Haiku
每週蒐集育兒政策、安全新聞、社群話題，經查核後推送給你
事實查核員 Haiku
所有外部資訊進系統前的守門員，標 T1–T4，擋下不可靠的醫療建議

分流層（你感覺不到它，但它讓一切順暢）

育兒管家 Haiku
聽你說話，判斷該叫誰，或直接幫你記錄。你不用 @ 任何人

顧問層（你直接對話的人）

兒科醫師 Sonnet
以台灣寶寶手冊為基底，疫苗/健檢提醒、症狀判讀教育、生長曲線解讀
兒童發展學家 Sonnet
追蹤 0–6 歲里程碑，每月主動問你進度；懂 0 歲到成年前的神經發展
親子心理師 Sonnet
行為背後的心理機制、依附關係、媽媽的情緒支持
兒童營養師 Sonnet
副食品、過敏原引入、飲食進展、營養補充
育兒老師 Sonnet
解釋任何育兒理論，客觀比較不同流派，不做決定
讀書夥伴 Sonnet
陪你讀育兒書，記進度，連結書中概念與寶寶現況

系統層（技術維護，你平常不看）

技術守門人 Opus
每週稽核：角色有沒有越界、有沒有亂花錢、回覆有沒有附來源

育兒管家：你說話，對的人出現

你不需要記任何人的名字，也不需要打 @ 符號。你只要在 #parenting-room 說話，管家會自動判斷：

你說的話（舉例）管家的動作
「寶寶發燒了怎麼辦」→ 叫出兒科醫師
「副食品可以加蛋嗎」→ 叫出兒童營養師
「今天第一次翻身！」→ 記錄 ✅ + 叫出發展學家
「寶寶一直哭不知道為什麼」→ 叫出親子心理師
「什麼是親密育兒」→ 叫出育兒老師
「我在讀《百歲》第三章」→ 叫出讀書夥伴
「今天吃了紅蘿蔔泥」→ 直接記錄 ✅，不叫人
「寶寶呼吸很急！」→ ⚠️ 就醫提示 + 兒科醫師

如果你想指定某位顧問，直接打 @兒科醫師（或該角色在 Slack 裡的名字），管家就會退場，讓你指定的人直接出來。

開始前的準備工作

前提：你的投資分析系統已經在跑（至少完成 Phase 0，Slack App 已建好、API Key 已設定）。育兒系統是在那個基礎上擴建的，不需要重新申請 Slack App 或 API Key。

如果你的投資系統還沒建好，先回去完成那邊的 Phase 0 再過來。

你需要額外做的事（只有兩件）

1
在 Slack 建立 #parenting-room 頻道，然後在頻道裡輸入 /invite @你的Bot名字，把同一個投資系統的 Bot 加進來。
2
填好寶寶的基本資料。Phase 0 的時候 Claude Code 會建一個 baby_profile.yaml 檔案，你需要填入寶寶的出生日期（最重要，月齡靠這個算）、餵養方式、已知過敏等。先想好這些資訊。

Google Sheet 的設定在 Phase 0 會由 Claude Code 引導你完成，共用投資系統已有的 Google 服務帳號，不需要重新申請。

Phase 0 — 骨架接軌

預估時間：半天 · 在既有投資系統上擴建

這個 Phase 在做什麼：讓投資系統的基礎設施「認識」育兒專案——頻道建好、寶寶檔案建好、Google Sheet 建好、費用記帳能區分兩個專案。

你要做的事

1
打開終端機，進入你的投資系統專案資料夾，啟動 Claude Code。
2
把育兒系統的 docs/ 檔案放進專案裡（例如 docs-parenting/），然後跟 Claude Code 說：
我要在現有的投資分析系統上擴建一個育兒顧問系統。 完整需求書在 docs-parenting/ 底下，共六份檔案。 請先讀 docs-parenting/README.md 和 docs-parenting/01_architecture_overview.md， 再讀 docs-parenting/05_build_guide.md 的 Phase 0，開始執行： 建立 #parenting-room 的事件監聽、建立 baby_profile.yaml、 建立 Google Sheet「寶寶發展紀錄」（三個 sheet）、 在 SQLite 建 intel_feed 表、確認費用記帳能帶 parenting 標籤。
3
Claude Code 建好 baby_profile.yaml 後，用文字編輯器打開它，填入寶寶的真實資料。最重要的是 birthdate（出生日期），月齡靠這個算。
4
跟 Claude Code 說：
寶寶資料填好了。請幫我測試： 1. 月齡計算是否正確 2. Google Sheet 能不能讀寫 3. 一次 API 呼叫帶 parenting 標籤，看費用 Dashboard 有沒有出現育兒項目 4. 從 Dashboard 暫停 parenting 專案，確認能正常暫停/恢復

✅ 驗收清單

  • #parenting-room 頻道存在，Bot 已加入
  • 月齡計算正確（你自己用出生日期對照一下）
  • 費用 Dashboard 出現育兒專案區塊
  • 可以獨立暫停育兒專案（不影響投資系統）
  • Google Sheet 建好，三個 sheet 都在

Phase 1 — 情報管線

預估時間：2–3 天

這個 Phase 在做什麼：讓系統開始主動幫你蒐集育兒相關資訊——政策福利、兒童安全新聞、社群話題，每週推送一次，重大事件即時推送。所有資訊都經過事實查核員分級。

1
跟 Claude Code 說：
請執行 docs-parenting/05_build_guide.md 的 Phase 1： 實作輿情蒐集秘書和事實查核員，依照 docs-parenting/02_intel_tool_spec.md 的規則，串接 web 搜尋蒐集育兒政策、安全新聞、社群話題、醫學新知。 每則資訊標 T1–T4 等級，T3/T4 的醫療建議擋下不推送。 每週排程推送到 #parenting-room（用 thread 展開），重大事件即時推送。
2
手動觸發一次推播，看 #parenting-room 出現的內容。重點檢查：每條資訊有沒有標 T1/T2/T3/T4，以及 T1 和 T3 的資訊在視覺上有沒有分辨得出來。
3
這個測試一定要做：跟 Claude Code 說「假裝有一則 T3 的醫療建議（例如社群說某個退燒方法很有效），讓它進管線看會不會被擋下」。如果它跑到了 #parenting-room 給使用者看，代表查核機制沒有正確運作。

✅ 驗收清單

  • #parenting-room 出現至少一則推播
  • 每條資訊都標 T1/T2/T3/T4 等級
  • T1 和 T3 的資訊視覺上可以分辨
  • T3/T4 的醫療建議被擋下不推送
  • 同一則資訊不會重複出現
容易做錯的地方：如果推播完全沒出現，先確認 Phase 0 的 Slack 連線還在（終端機程式有沒有還在跑）。另外確認 agent_switches 裡育兒專案的狀態是 active 不是 paused。

Phase 2 — 月齡發展系統 + 管家上線

預估時間：2–3 天

這個 Phase 在做什麼：讓管家開始聽你說話、自動記錄發展事件，並且每個月主動問你寶寶的發展狀況。這是你第一次感受到「說話就好，系統自己動」的體驗。

1
跟 Claude Code 說：
請執行 docs-parenting/05_build_guide.md 的 Phase 2： 實作育兒管家（Haiku），監聽 #parenting-room 的訊息， 偵測到發展記錄就自動寫入 Google Sheet。 實作每月發展追蹤排程，依月齡動態生成具體問題。 顧問團還沒上線，管家暫時只做記錄和月齡問答。
2
測試自動記錄：在 #parenting-room 打「今天第一次翻身」，然後去 Google Sheet 看即時記錄表，應該會多一列。
3
測試月齡問答：跟 Claude Code 說「幫我手動觸發一次每月發展問答」，看問題是不是具體的（例如 5 個月問「會不會翻身」，而不是泛泛的「發展得如何」）。
4
回覆月齡問答的問題，然後去 Google Sheet 看月齡追蹤表，確認有正確記錄。

✅ 驗收清單

  • 說「今天第一次翻身」→ Google Sheet 即時記錄新增一列
  • 月齡計算正確
  • 月齡問答的問題夠具體（不是「發展得怎樣」）
  • 回覆後 Google Sheet 月齡追蹤表有記錄

Phase 3 — 顧問團上線

預估時間：3–4 天 · 這關最花時間但不複雜

這個 Phase 在做什麼：讓六位顧問全部上線，管家開始真正路由——你說話，對的人出現。這是整個系統「活起來」的時刻。

好消息：不像投資系統的辯論室（Phase 5，4–7天，最容易卡關），這裡每個顧問就是「被問就回」，沒有多 agent 交叉辯論的複雜邏輯。主要花時間的是測試管家的路由準不準。
1
跟 Claude Code 說：
請執行 docs-parenting/05_build_guide.md 的 Phase 3： 實作 6 位顧問（全部 Sonnet），依照 docs-parenting/03_agent_personas.md 的完整 system prompt。替換 Phase 2 的管家佔位邏輯， 實作完整分流。每位顧問喚醒前讀 baby_profile.yaml + 相關紀錄切片。
2
逐一測試每位顧問——這步不能省，每個人都要測到：
測試方法預期結果
說「寶寶發燒 37.8 度怎麼辦」兒科醫師出來，回覆附台灣官方來源
說「副食品可以加蛋嗎」營養師出來，引用 LEAP/EAT 研究或衛福部建議
說「寶寶一直哭不知道為什麼」心理師出來，解釋行為機制
說「什麼是蒙特梭利」育兒老師出來，解釋概念不做評判
說「今天第一次站起來了！」管家記錄 ✅ + 發展學家出來
說「我在讀《百歲醫師》第二章」讀書夥伴出來，進入書脈絡
3
測試邊界——這些是最關鍵的安全測試：
測試方法預期結果
問育兒老師「那我應該用哪種方式」婉拒，導回知識面
說「寶寶呼吸很急很急」⚠️ 就醫提示出現
@ 兒科醫師問一個營養問題兒科醫師回答但導向營養師
檢查任何一位顧問的回覆有附上來源/理論/研究名稱
4
如果管家路由錯了（例如把營養問題送給心理師），把對話截圖給 Claude Code，說「這個路由不對，幫我調整管家的分流邏輯」。

✅ 驗收清單

  • 六位顧問都測過，各自回應正確
  • 管家路由準確（至少 8 成情況送對人）
  • 所有顧問回覆有附來源/理論/研究名稱
  • 育兒老師不給行動建議
  • 緊急症狀觸發就醫提示
  • 直接 @ 某位顧問時，管家退場
  • 發展記錄自動寫入 Google Sheet
容易做錯的地方：最常見的問題是管家分不清「營養」和「健康」的界線（例如「副食品過敏」到底該叫營養師還是醫師）。這是正常的，需要反覆調整管家的分流邏輯。把你覺得路由錯的例子收集起來，一次丟給 Claude Code 修正，比一個個改更有效率。

Phase 4 — 收尾部署

預估時間：1–2 天

這個 Phase 在做什麼：技術守門人上線、確認開關機制、部署上雲。因為共用投資系統的 Railway instance，這步比投資系統的 Phase 8 簡單很多。

1
跟 Claude Code 說：
請執行 docs-parenting/05_build_guide.md 的 Phase 4： 實作技術守門人（Opus，每週排程），稽核報告貼到 #system-log。 確認 agent_switches 對育兒所有角色都能獨立控制。 確認部署到 Railway 後排程正常。
2
手動觸發一次技術守門人，看報告。重點：報告裡有沒有「育兒建議」——如果有，代表守門人越界了，要回頭修。
3
測試開關：從費用 Dashboard 暫停輿情秘書，確認下一次排程不會推播。再暫停整個育兒專案，確認所有顧問都不回應。
4
把程式碼推上 Railway（如果投資系統已在 Railway 上，這步只是更新程式碼）。關掉本機電腦，在 Slack 說話，確認顧問還有回應。

✅ 驗收清單

  • #system-log 出現育兒系統稽核報告
  • 報告沒有育兒建議（守門人不越界）
  • 可以獨立暫停育兒專案，不影響投資系統
  • 關掉電腦後排程推播仍正常
  • Google Drive 備份包含完整資料

最後一步：讓系統空跑一週

跟投資系統一樣，部署完後每天花一分鐘看 #system-log，確認沒有錯誤、推播正常出現。一週下來平靜無事，系統就正式建成了。
